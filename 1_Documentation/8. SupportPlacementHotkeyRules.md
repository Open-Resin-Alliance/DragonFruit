# Support Placement Hotkey Rules

## Purpose
This document defines the intended user-facing rules for support placement hotkeys, including:
- hotkey precedence
- first-click-dependent behavior
- how support type is chosen
- cancellation and reset behavior
- behavior that is explicitly not allowed

This document is intended to be the reference for placement behavior discussions and bug fixing.

## Core Principle
Support placement is determined by two things together:
- which hotkeys are currently held
- what valid target is clicked first

Hotkeys do not always map one-to-one to a single support type.
Some hotkeys select an interaction family, and the first valid click determines which placement flow is entered.

## Held-Key Meanings
### No hotkey held
- Default support placement behavior
- Normal trunk/root-style placement flow

### Hold `Alt`
This enters the Alt placement family.
What happens next depends on the first valid click.

### Hold `Ctrl`
This enters the Ctrl placement family.
This is used for kickstand placement.

### Hold `Ctrl+Alt`
This enters the leaf placement family.
This combination is more specific than either `Ctrl` alone or `Alt` alone.

## Hotkey Precedence Rules
### Specific combination wins
If multiple placement hotkeys overlap, the more specific combination must win.

Precedence order:
1. `Ctrl+Alt` -> Leaf
2. `Alt` -> Branch/Brace family
3. `Ctrl` -> Kickstand
4. No hotkey -> Default placement

### Key order does not matter
The order that keys are pressed must not matter.
Only the final currently-held modifier state matters.

These must behave identically:
- hold `Alt`, then hold `Ctrl`
- hold `Ctrl`, then hold `Alt`

If `Ctrl` and `Alt` are both held when the user clicks the mesh first, the interaction is leaf placement.

## First-Click-Dependent Rules
## `Alt` family rules
When only `Alt` is held, the first valid click determines which Alt-based flow is entered.

### `Alt` + first click on model
This enters the branch-style flow.
- first model click sets the tip/contact point
- second action determines final outcome

Second action outcomes:
- second click on support shaft -> Branch
- second click on model -> Twig or Stick

### `Alt` + first click on support
This enters the brace-style flow.
- first support click selects the start
- second support click completes the brace

## `Ctrl` rules
### `Ctrl` + first click on support
This is kickstand placement.

`Ctrl` alone is not leaf placement.

## `Ctrl+Alt` rules
### `Ctrl+Alt` + first click on model
This is leaf placement.
- first model click sets the leaf contact/tip
- second click on support shaft commits the leaf

This must always be leaf placement regardless of whether `Ctrl` or `Alt` was pressed first.

## Support Type Resolution Matrix
### No hotkey held
- first click on model -> default trunk/root-style placement

### Hold `Alt`
- first click on model -> branch-style flow
- second click on support -> Branch
- second click on model -> Twig or Stick
- first click on support -> brace-style flow
- second click on support -> Brace

### Hold `Ctrl`
- first click on support -> Kickstand

### Hold `Ctrl+Alt`
- first click on model -> Leaf flow
- second click on support -> Leaf

## Interaction Ownership Rules
### Leaf must override kickstand
When `Ctrl+Alt` is held, leaf placement owns the interaction.
Kickstand must not steal support clicks from an active leaf flow.

### Leaf must override Alt-only branch behavior
When `Ctrl+Alt` is held, leaf placement owns the interaction.
Alt-only branch behavior must not win over leaf.

### Alt family behavior depends on first click
For Alt-only interactions:
- model-first means branch-style flow
- support-first means brace-style flow

## Cancellation Rules
### Branch cancellation
Releasing `Alt` cancels branch placement immediately.

### Leaf cancellation
Releasing either required key cancels leaf placement immediately.
That means releasing either:
- `Ctrl`
- `Alt`

must cancel leaf placement.

### Kickstand cancellation
Releasing `Ctrl` cancels kickstand placement immediately.

### Blur / missed keyup fallback
If the browser misses a keyup because focus changed, fallback detection must still cancel the placement mode.
Placement modes must not remain stuck active after a key is no longer held.

## Reset Rules on Cancel
When a hotkey placement mode is canceled, all transient placement state must be cleared.

That includes:
- hover dot state
- preview geometry
- snap target state
- staged tip/base/contact state
- transient local cached hover/snap refs used to rebuild preview

Canceling a placement flow must not leave a persistent preview on screen.

## Re-entry Rules
Pressing the hotkey again after canceling must start from a fresh placement state.
Old preview state must not be reused or redisplayed automatically.

Examples of forbidden behavior:
- releasing the hotkey hides the preview, then pressing the hotkey again resurrects the old preview
- placement state remains active after a required key is released
- the result depends on whether `Ctrl` or `Alt` was pressed first when the final held state is the same

## Explicitly Forbidden Outcomes
The following behaviors are incorrect:
- `Ctrl+Alt`, click model first, and the app enters kickstand behavior
- `Ctrl+Alt`, click model first, and the app enters Alt-only branch behavior
- `Ctrl` then `Alt` behaves differently from `Alt` then `Ctrl` when the final held state is the same
- releasing a required key leaves the placement preview active
- pressing the hotkey again revives stale preview state from a previous placement attempt

## Short Reference
- `Alt` + model first -> Branch/Twig/Stick family
- `Alt` + support first -> Brace
- `Ctrl` + support first -> Kickstand
- `Ctrl+Alt` + model first -> Leaf
- `Ctrl+Alt` beats `Ctrl`
- `Ctrl+Alt` beats `Alt`
- key order does not matter
- releasing a required key cancels immediately
- canceling clears all transient placement state
- re-entering starts from a fresh state
