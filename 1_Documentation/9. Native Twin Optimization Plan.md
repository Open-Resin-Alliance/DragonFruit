# Native Twin Optimization Plan (Post-Beta)

## Status

- **Priority:** Next major optimization after stable beta
- **Scope:** Reduce slicing/export latency by minimizing frontend→backend bulk geometry IPC
- **Decision:** Move to a **native twin + diff synchronization** architecture

---

## Why This Exists

Current slicing pipeline still spends substantial time moving and staging large mesh buffers across IPC, even with raw payload transport and chunk tuning.

Observed bottleneck pattern:

- High stage-mesh IPC duration
- Low-to-moderate effective transport throughput under large scenes
- Significant bridge roundtrip wall time relative to native render/encode time

This plan addresses that by changing architecture, not just tuning chunk size.

---

## Product Constraints (Non-Negotiable)

1. **Frontend support editing must remain smooth** (drag/update interaction quality preserved).
2. **Support placement fidelity must remain exact** (knot/joint/segment positions match export).
3. **Stable beta work ships first** (this plan starts afterward).

---

## High-Level Architecture Target

### Core Idea

Treat Rust as a **native digital twin** of the build scene.

- Frontend owns real-time interaction and immediate preview feedback.
- Backend owns canonical slice-ready state for export/slicing.
- Frontend sends **small state diffs**, not triangle-heavy geometry streams.

### Asset Ownership

- Model meshes are loaded/cached natively by asset identity (path/hash), not resent repeatedly over IPC.
- Support system synchronizes as a graph/state snapshot + diffs (IDs + exact resolved coordinates/params), not support triangle payloads.

---

## Data Boundary Design

## Scene Twin Command Channel (Frontend → Rust)

Use a compact command stream (batched when possible):

- `scene.load_model`
- `scene.set_transform`
- `scene.remove_model`
- `supports.snapshot_begin/snapshot_chunk/snapshot_commit`
- `supports.apply_diff_batch`
- `scene.set_slice_settings`

### Support Diff Requirements

Every support-node diff carries:

- Stable support entity ID
- Parent relationship IDs (where applicable)
- Exact world-space resolved positions
- Exact diameter/radius/profile data
- Revision number + timestamp/sequence

This preserves the frontend-authored exactness for knot/joint positions.

## Twin Acknowledgement Channel (Rust → Frontend)

- `twin.revision_applied`
- `twin.out_of_sync`
- `twin.model_loaded`
- `twin.supports_applied`
- `twin.ready_for_slice`

Optional:

- `twin.preview_patch` (lightweight authoritative correction overlays)

---

## Fidelity Strategy (Supports Must Be 1:1)

### Manual Support Editing

- Frontend remains edit-authority during interaction.
- Rust mirror applies exact resolved support graph diffs.
- Slicing/export is gated on revision parity:
  - `frontendRevision == twinRevision`

### Auto-Generated Supports

Either:

1. Generate in Rust and preview Rust result in frontend, or
2. Share deterministic generation logic and validate by hash.

### Validation Gates

- Per-support-subtree hash (structure + resolved coordinates + diameters)
- Whole-scene twin hash
- Export blocked when hash mismatch exceeds tolerance policy

## Phased Implementation Plan

## Phase 0 — Prerequisites (After Beta Freeze)

**Goal:** Prepare rollout safely.

- [ ] Freeze current slicing transport behavior behind a feature flag baseline
- [ ] Add clear telemetry for:
  - scene load wall time
  - support sync time
  - twin apply time
  - slice start readiness delay
- [ ] Define rollback toggle to legacy mesh-staging path

**Exit Criteria:** We can compare new architecture against a stable baseline quickly.

## Phase 1 — Native Model Twin (No Supports Yet)

**Goal:** Remove repeated frontend mesh uploads for models.

- [ ] Introduce model asset registry in Rust (path/hash keyed)
- [ ] Add model instance twin commands (load/reuse/remove/set transform)
- [ ] Update scene import flow to call `scene.load_model` early
- [ ] Reuse already-loaded meshes for repeated exports/slices

**Exit Criteria:** Model geometry no longer transferred repeatedly over IPC for slicing.

## Phase 2 — Support Graph Snapshot Sync

**Goal:** Sync supports as state graph snapshots, not triangles.

- [ ] Define canonical support graph schema for IPC snapshot/diff payloads
- [ ] Implement chunked snapshot upload on scene-load for support-heavy files
- [ ] Implement `supports.snapshot_commit` validation + revision ack
- [ ] Keep frontend preview unchanged initially

**Exit Criteria:** Support geometry is no longer sent as mesh triangles during normal scene load path.

## Phase 3 — Incremental Support Diffs (Interactive)

**Goal:** During edits, send only changed support graph deltas.

- [ ] Add diff batching/coalescing window (e.g., 50–150ms)
- [ ] Stream diffs while dragging; throttle where needed
- [ ] Add `twinRevision` parity gating before slice/export
- [ ] Add out-of-sync recovery action (fast resnapshot)

**Exit Criteria:** Interactive support editing stays smooth and slice readiness remains deterministic.

## Phase 4 — Fidelity Hardening

**Goal:** Guarantee 1:1 correctness.

- [ ] Add per-subtree hash and full-scene hash checks
- [ ] Add pre-export assert: frontend/twin revisions equal
- [ ] Add mismatch telemetry and auto-repair trigger
- [ ] Add deterministic tests for support graph replay

**Exit Criteria:** “What user sees” equals “what twin slices” within agreed tolerance.

## Phase 5 — Legacy Path Decommission

**Goal:** Remove high-cost geometry staging for primary flows.

- [ ] Keep legacy triangle-staging as emergency fallback behind flag
- [ ] Migrate all stable flows to twin model
- [ ] Remove obsolete IPC transport code in staged cleanup PRs

**Exit Criteria:** Twin architecture is default and stable.

## IPC Contract Sketch (Draft)

```ts
type TwinRevision = number;

type LoadModelCmd = {
  type: "scene.load_model";
  modelId: string;
  assetPath: string;
  assetHash: string;
  transform: {
    position: { x: number; y: number; z: number };
    rotation: { x: number; y: number; z: number };
    scale: { x: number; y: number; z: number };
  };
};

type SupportDiff = {
  id: string;
  kind:
    | "root"
    | "knot"
    | "joint"
    | "segment"
    | "contact"
    | "branch"
    | "trunk"
    | "brace"
    | "kickstand";
  op: "add" | "update" | "remove";
  parentId?: string;
  resolvedWorld?: { x: number; y: number; z: number };
  params?: Record<string, unknown>;
};

type ApplySupportDiffBatchCmd = {
  type: "supports.apply_diff_batch";
  baseRevision: TwinRevision;
  targetRevision: TwinRevision;
  diffs: SupportDiff[];
};
```

## Performance Targets

Post-migration targets (initial):

- Reduce stage transport wall time by **>60%** on large support-heavy scenes
- Reduce bridge overhead contribution to total wall by **>50%**
- Keep interactive support edit latency under current beta baseline
- Keep export/slice start gating under **250 ms** in synced state

---

## Risk Register & Mitigations

1. **Drift between frontend and twin state**
   - Mitigation: strict revision IDs, hash checks, resnapshot recovery path.

2. **Support schema churn during active feature work**
   - Mitigation: versioned schema + adapters.

3. **User-perceived lag from background sync**
   - Mitigation: diff coalescing, async acks, non-blocking UI updates.

4. **Regression risk near release branch**
   - Mitigation: feature flags, shadow mode, phased rollout.

---

## Rollout Plan

1. **Dev-only feature flag** (`nativeTwinV1`)
2. **Shadow mode**: apply twin updates while still slicing via legacy path for verification
3. **Canary mode**: selected users/projects use twin slicing path
4. **Default-on** after stability + telemetry threshold met

---

## Definition of Done

This initiative is done when all are true:

- Slicing no longer depends on bulk geometry IPC from frontend for normal model/support flows.
- Support fidelity tests prove 1:1 parity for manual edit scenarios.
- Performance targets are met on representative heavy scenes.
- Legacy fallback exists but is not default.

---

## Immediate Next Step (After Beta)

Start with **Phase 1 + Phase 2 schema definition** in one planning PR:

- Rust twin command endpoints (model registry + support snapshot ingestion)
- Frontend twin-sync adapter scaffolding
- telemetry hooks for migration metrics

This yields visible risk reduction quickly while keeping implementation incremental.
